# 设计上下文包(Project Context Pack)

> 这是一份**正在被讨论填充**的设计上下文包。任何人 / agent 读完它,应能创建第一个实现任务,无需回溯历史对话、无需补上下文、无需猜哪个文档权威。
> 它由 `project-context-pack` skill 通过和用户一轮一轮的持续讨论,逐层沉淀而成。

## 读法总图(七层,各管什么)

| 层 | 管什么 | 何时读 |
|---|---|---|
| `00-goal-boundary.md` | 北极星 / 设计箴言 / Done / 非目标 / 约束 / 停止条件 / 自主边界 | 所有人,最先读 |
| `10-foundation/` | 是什么 / 为什么(产品问题、领域模型、架构、词汇表) | 判断"该不该这么做" |
| `20-contracts/` | 必须怎么做(可被 reviewer / CI 执行的约束) | 实现 / review / 写 gate |
| `30-roadmap/` | 什么时候做 + 验收状态(里程碑、decision-log、ADR) | 判断当前该做什么 |
| `40-task-packets/` | 可派工的任务包(inputs/outputs/forbidden/gates/stop) | 派工前 |
| `60-diagrams/` | 图(非规范,辅助;每张须暴露一个未解张力,否则不画) | 需要图建立心智模型 |
| `90-evidence/` | 证据 / 调研 / 讨论留痕(非规范,需"晋升"才成决策源) | 追溯某结论为何存在 |

各层**单向依赖**:上层定义,下层只能落实,不能反向改写。下层发现上层不对,回上层显式改 + 记录,不在下层偷偷绕过。

## 权威优先级表(冲突时谁让谁)

```
ADR > Decision Log【硬】 > Contract(20) > Foundation 正文(10)
    > Goal Boundary(00) > Roadmap(30) > Task Packet(40) > 图/证据(60,90)
```

两份文档打架:高优先级胜,**当场 patch 低优先级文件至一致**,不在对话里口头绕过。若认为高优先级本身过时,走风化复核或新建 ADR 推翻,不能直接改高优先级正文。
