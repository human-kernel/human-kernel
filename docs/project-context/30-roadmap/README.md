# 30 · Roadmap(路线层)读法

> 本目录回答**什么时候做 + 验收状态**:里程碑、依赖、决策记录。它安排先后和验收口径,但**不能重定义 foundation 概念**。
> 下面是导航地图。

## 目录结构

| 文件 / 目录 | 管什么 |
|---|---|
| `decision-log.md` | 决策流水(每条【硬/定/开】+ 理由 + 出处)+ 被否决方案表 + 权威优先级 + 开放项 |
| `milestones/` | 每个里程碑一个子目录;先复制 `_TEMPLATE/` 再改名填充 |
| `adr/`(按需建) | 不可变架构决策(满足三门槛才写,见 decision-log 说明) |

## 里程碑怎么定义(核心)

每个里程碑结束时,**系统必须处于某个可用状态** —— 不是"写了一堆代码",而是"现在能做一件原来做不了的事"。

新建里程碑:复制 `milestones/_TEMPLATE/` 为 `milestones/m1-<名字>/`,填充两个文件:

- **`00-overview.md`** — North Star(本里程碑一句话目标)/ In Scope / Non-goal(本里程碑明确不做)/ 入口条件 / 验收标准(可勾选)/ 依赖。
- **`02-status-checklist.md`** — 功能点分组 + 逐项处置状态 + 证据。

里程碑按**技术依赖**严格排序;跨里程碑依赖在 `00-overview` 的"依赖"节标注。

## 铁律

- 里程碑的 Non-goal 和验收标准必须可观察,不能是"做好 X 模块"这种内部说法。
- 不要在这里重定义 foundation 概念;冲突时 foundation 正文优先(见 README 权威优先级表)。
- 验收标准未全绿的里程碑不算完成。
