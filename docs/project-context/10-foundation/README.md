# 10 · Foundation(定义层)读法

> 本目录回答**系统是什么、为什么**:产品问题、核心概念、架构、词汇。它不记录进度(进度在 30-roadmap),也不是可执行约束(那在 20-contracts)。
> 下面是导航地图。讨论清楚一块就在这里新建对应文件;没讨论到就别建空壳。

## 典型文件(按需建,不强求全建)

| 文件 | 管什么 | 不管什么 |
|---|---|---|
| `product-problem.md` | 为谁、解决什么、典型场景、价值 | 实现切片、CI 命令 |
| `domain-model.md` | 核心概念、实体、关系、关键状态 | CLI 参数、外部 API 细节 |
| `system-architecture.md` | 分层、模块边界、数据流、状态归属 | 逐文件目录拆分 |
| `glossary.md` | 规范命名(同一词不指两个东西) | 新决策(去 decision-log) |

## 铁律

- **这层一旦定下,实现层(20/40)只能落实,不能反向改写。** 改概念要回这里显式改 + 记 decision-log。
- **同一概念在 foundation / glossary / contracts 必须一致。** 比如某状态机有几个值,这几处不能各说各话(这是最常见的自洽 bug)。
- 新概念先进 decision-log 或 ADR,再回写本层正文。
- 不要把"临时原型看起来这样"写成 foundation 事实;原型结论需先晋升。
